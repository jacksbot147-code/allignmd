# AlignMD

*Precision Matching for Modern Healthcare*

A two-sided healthcare staffing platform — clinicians (NP/PA/MD/DO/CRNA)
matched to facility jobs by a credential-aware, rule-based match score.

Full plan: see `../AlignMD-Build-Plan.md`. Built as a Day14 segment.

---

## Status — Phase 6 complete

**Phase 0 — security foundation: done.** Nine Supabase migrations:
`0001_schema` (~18 tables), `0002_audit` (audit-log triggers on sensitive
tables), `0003_rls` (row-level security, role-gated), `0004_app_bootstrap`
(auth-user → app_users trigger + private document bucket + storage policies),
`0005_phase1_finish` (allied-health roles, provider archiving, SSN moved to a
privileged-only `provider_private` table), `0006_phase3` (application_responses
edit-tracking + one-per-provider index, provider_references timestamps + audit
trigger), `0007_portals` (facility_contact → facility link + self-service RLS
for provider and facility-contact roles), `0008_phase5` (license_applications
provenance / lifecycle columns, indexes, and a partial unique index — one live
application per clinician + state), `0009_phase6` (`outreach_drafts` — a
staff-only log of generated draft copy, with `staff_all` RLS and indexes).

**Phase 1 — Provider CRM core: substantially built.**
- Email/password auth; first account becomes workspace admin.
- Providers — list (filterable), create, detail (4 tabs), edit.
- Credentials — add / verify / delete, plus a `/credentials` expiry tracker
  with 30/60/90-day buckets.
- Documents — upload to a private Storage bucket, short-lived signed-URL
  download, delete.
- Activity log — per-provider timeline.
- Pipeline board — 7 stages (New → Screen → Credentialing → Submitted →
  Interview → Offer → Placed).
- Dashboard — KPIs, expiring credentials, pipeline snapshot, recent activity.

### Ops agents (`scripts/`)

- **`credential-expiry-alerter.mjs`** — scans credentials and opens a task
  reminder for each expiring within 90 days (buckets: expired / 30 / 60 / 90,
  deduped). This is what keeps the dashboard's task view live. Run daily:
  `node scripts/credential-expiry-alerter.mjs`.

### Roles

`admin`, `recruiter`, `credentialing_coordinator`, `provider`,
`facility_contact`. Staff (admin / recruiter / coordinator) run the CRM;
restricted data (malpractice, IDs) is limited to admin + coordinator. Every
mutating server action calls `requireStaff()`; RLS is the database backstop.

### Decision taken: SSN

Per the build plan, **the full SSN is not stored** — the schema keeps only
`ssn_last4`. Collect the full value just-in-time when a background-check
vendor needs it; never persist it.

## Setup

`.env.local` is generated by the Day14 `supabase-provisioner` (project URL +
anon key + service-role key). Then:

1. **Run the migrations** — Supabase SQL editor, run `supabase/migrations/`
   in order (0001 → 0008). Or `supabase db push`.
2. **Install + run** — `npm install`, then `npm run dev` → http://localhost:3000.
3. **Schedule the alerter** — run `scripts/credential-expiry-alerter.mjs`
   daily (cron / launchd) so credential reminders stay current.

## Phase 1 — complete

- **Input validation** — `src/lib/validation.ts`. NPI is checked with the
  CMS Luhn algorithm, `ssn_last4` must be 4 digits, credential states are
  validated against the US state list, and issue/expiry dates are sanity-
  checked (valid calendar dates, expiry not before issue). Server actions
  reject bad input and surface a readable message.
- **True field-level SSN protection** — the SSN no longer lives on
  `providers`. It moved to `provider_private`, a separate table that only
  privileged staff (admin / credentialing coordinator) can read or write,
  enforced by RLS *and* the audit trigger. A normal `select *` on a provider
  can never return it.
- **Provider availability** — full add / remove UI on a new Availability tab
  of the provider detail page.
- **Provider archive / restore** — soft archive (never a hard delete, so the
  audit trail is preserved). Archived providers drop out of the list,
  pipeline board, and dashboard counts; an Archived view brings them back.
- **Allied-health roles** — provider types now include PT / OT / SLP
  alongside NP / PA / MD / DO / CRNA.

Decisions for Phase 2 (confirmed with the client): everyone logs in
(providers + facility contacts get accounts), background checks go through a
vendor integration, compact / IMLC licenses count for every member state,
matching is advisory with severity tiers rather than hard filters, and the
primary screen is a job with its ranked list of matching clinicians.

## Phase 2 — complete (MVP)

Facilities, jobs, and the rule-based match score. End of Phase 2 is a
working MVP.

- **Facilities & jobs** — facility profiles; job profiles with rate cards
  (hourly / callback / OT / weekend / holiday) and structured requirements
  (license states, required certs, minimum experience).
- **Match engine** — `src/lib/match.ts`, a weighted rule-based score (0–100)
  across licensure, specialty, certifications, experience and location.
  Advisory: every clinician is scored and shown; serious gaps cap the tier.
  Compact / IMLC / PT-Compact licenses count for every member state.
- **Job → ranked clinicians** — the primary screen: a job with every active
  clinician scored and ranked, plus a submissions pipeline.
- **Reverse match** — the provider detail page (Overview tab) scores that
  clinician against every open job and lists the best-fit roles with tier
  badges linking through to each job.
- **Dashboard** — now also surfaces open-jobs and submissions KPIs and a
  newest-open-jobs panel.
- **Bulk importer** — `/import`: staff upload a CSV of clinicians and/or a
  CSV of facilities. CSV is parsed server-side by a dependency-free parser
  (quoted fields, embedded commas/newlines, escaped quotes, BOM, CRLF); every
  row is validated independently; valid rows are inserted and rejected rows
  are reported per-row with the reason. Column templates are downloadable.

## Phase 3 — complete

Provider intake & onboarding — the application survey, structured references,
and a printable CV view.

- **Application survey** — a five-section intake form (professional profile,
  work history, education & training, assignment preferences, screening
  disclosures) on a new **Application** tab of the provider detail page. The
  whole survey round-trips through `application_responses.payload` (jsonb);
  `src/lib/application.ts` gives that free-form payload a stable, fully-typed
  shape. The form has a draft/submitted lifecycle — *Save changes* persists a
  draft, *Mark as submitted* stamps `submitted_at`, *Reopen as draft* clears
  it — plus a live completeness percentage. Work history (up to 4 positions)
  and education (up to 3 entries) use fixed slots parsed server-side.
- **References** — a new **References** tab: add / list / edit / mark-verified
  UI over `provider_references`, structured for primary-source reference
  calls (name, contact, relationship, verified flag, call timestamp, notes).
  Marking verified records the call time; references carry an audit trail.
- **CV summary** — `/providers/[id]/cv`, a clean, printable CV-style page that
  assembles the provider record, licenses & certifications (malpractice
  excluded), application work history & education, assignment preferences, and
  references into one document. A print stylesheet drops the app chrome so it
  prints / saves to PDF cleanly. Reachable from the provider header and the
  Application tab.

All Phase 3 mutations stay behind `requireStaff()`; RLS (`staff_all` policies
from `0003`) is the database backstop. `tsc --noEmit` and `next lint` are
clean.

## Phase 5 — complete

The state-license application assistant — a survey-driven helper that reduces
the licensing headache when a clinician needs to get licensed in a new US
state. Honest scope (per the build plan): it **pre-fills and organizes** an
application; it does not auto-submit. Every state board still receives its own
submission.

- **License applications** — a top-level **`/licensing`** page lists every
  in-flight application across the clinician roster, with KPIs (in-flight,
  draft, submitted, issued), a status filter, and a start-new form. Each
  provider detail page also gains a **Licensing** tab listing that clinician's
  applications. Status tracks the lifecycle **draft → submitted → issued**
  (plus `withdrawn`); `submitted_at` / `issued_at` are stamped on transition.
- **Per-application workspace** — `/licensing/[id]` carries the status
  controls, the checklist, and the wizard for one application.
- **State checklist** — `src/lib/licensing.ts` → `licenseChecklistForRole()`
  builds the items a state application typically needs, varied sensibly by
  role: primary-source license verification, NPI, board / national
  certification (wording adapts per discipline), education verification,
  background check, malpractice history, work history, photo ID and fee for
  everyone; a **DEA / state CSR** item for prescribers (NP / PA / MD / DO /
  CRNA); a **collaborative / supervisory agreement** item for NP and PA. Each
  item carries a completion status, a working note, and a profile-derived
  hint (e.g. "NPI on file…", "2 of 3 state licenses verified").
- **Wizard** — a five-step survey (`LicenseWizard` client component): applicant
  identity, education & training, licensure & certification, practice history,
  disclosures & attestations. The whole survey plus the checklist state lives
  in `license_applications.document_bundle` (jsonb); `src/lib/licensing.ts`
  gives that free-form column a stable, fully-typed `LicenseBundle` shape.
- **Pre-fill** — `prefillSurveyFromProvider()` populates the wizard from the
  clinician's **structured** data: the provider record, their credentials
  (state licenses, DEA, board certs), and their intake application. A blank
  application opens pre-filled for staff to review rather than retype.
  Extracting data from **uploaded documents** (scanned licenses, DEA
  certificates) is deliberately stubbed behind the `DocumentExtractor`
  interface with a clear `TODO` — `noopDocumentExtractor` is a typed no-op and
  `applyExtractedFields()` is the ready merge path. No real OCR is attempted.
- **Migration `0008_phase5`** — adds `created_by`, `submitted_at`, `issued_at`
  to `license_applications`, plus provider / status indexes and a partial
  unique index (one live application per clinician + state; a `withdrawn` one
  does not block re-applying). RLS reuses the `staff_all` policy from `0003`
  and the audit trigger from `0002` — no new row-visibility rules.

Phase 5 mutations live in `src/app/(app)/licensing/actions.ts`, all behind
`requireStaff()`. `tsc --noEmit` and `next lint` are clean.

## Phase 6 — complete

Reporting, comms & scale.

- **Reports** — a top-level **`/reports`** page (in the sidebar). KPI cards for
  total submissions, placements, **fill rate** (placed ÷ submitted), **average
  time-to-fill** (mean days from a job's `created_at` to a submission reaching
  the `placed` stage), and open jobs. Below them, a **submissions-by-pipeline-
  stage** bar breakdown and two tables — **by recruiter** (the `owner_id` of
  each submitted clinician) and **by facility** (the facility behind each job)
  — each showing submissions, placements and fill rate. The aggregation is a
  pure module, `src/lib/reports.ts` (`buildReport`, `timeToFillDays`,
  `formatPercent`); the page only does the Supabase reads and the rendering.
- **Outreach drafts** — a top-level **`/outreach`** page (in the sidebar) that
  generates **email and SMS draft copy** for one clinician or a shortlist.
  Pick a job for context (the page ranks every active clinician with the match
  engine) or run a general check-in, tick the clinicians, and AlignMD writes an
  email draft (subject + body) and an SMS draft for each. **Draft-only by
  design** — nothing is sent, no messaging provider is connected, and there are
  no credentials anywhere near it; the copy is displayed for copy/paste (each
  draft has a Copy button) and logged to `outreach_drafts`. Draft text is built
  by the pure `src/lib/outreach.ts` (`buildEmailDraft` / `buildSmsDraft`);
  generation and removal live in `src/app/(app)/outreach/actions.ts`, behind
  `requireStaff()`. Migration `0009_phase6` adds the `outreach_drafts` table.
- **Search & scale hardening** — the providers and jobs lists are now
  **server-side paginated** (`src/lib/pagination.ts`, `PAGE_SIZE` 25). Each
  list runs a cheap `count` query, then fetches only the requested page with
  `range()`; existing filters (provider search / role / stage / archived, job
  status) are applied to both queries and preserved across page links by the
  shared `Pagination` component. Both list queries select **only the columns
  the table renders** — the jobs list dropped its `select("*")`, and its
  per-job submission counts are now fetched only for the jobs on the current
  page (`in(...)`) instead of loading the whole `submissions` table.

Phase 6 mutations live in `src/app/(app)/outreach/actions.ts`, all behind
`requireStaff()`. `tsc --noEmit` and `next lint` are clean.
